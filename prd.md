产品需求文档（PRD）
产品名称：街舞教课日志（暂定名）
版本：v1.0（MVP自用版）
文档输出日期：2026-07-07
开发者模式：纯本地离线应用，零云端依赖

1. 产品目标与范围
核心痛点：解决多工作室排课混乱、月底手工对账耗时、停课补偿计算易遗漏的问题。

MVP边界：只做“记录+计算”，不做“提醒推送”（推送需后端，复杂且非刚需）。只做“日历查看”和“列表对账”两个核心Tab。

成功标准：月底对账时间从手动半小时缩减至打开App点开对账Tab查看汇总数据的10秒以内。

2. 用户角色
唯一角色：你（街舞老师）。无多用户、无分享、无登录。

3. 核心数据实体定义（决定数据库设计）
实体名称	核心字段（仅列出关键约束）
工作室	id, 名称, 单节课时费(元), 停课补偿比例(0~1), 固定发薪日(1-31整数) , 地址(文本)
课程模板	id, 关联工作室id, 类型(常规/临时), 星期几(1-7), 开始时间(HH:mm), 结束时间, 课程内容标签(文本), 建议出门提前分钟数
课程实例	id, 关联模板id(可为空), 具体日期(YYYY-MM-DD), 开课状态(待开/已开/停课), 收款状态(未收/已收), 实际应收金额(浮点数) , 当日音乐备注(文本)
约束规则：开课状态与收款状态完全解耦，可任意组合（停课+已收、已开+未收均可共存）。

4. 功能需求详情（按页面/模块）
4.1 模块一：工作室管理（底层配置）
入口：侧边栏或设置页。

操作：增、删、改、查。

关键交互：删除工作室时，弹窗二次确认，并提示“将同时删除该工作室所有历史课程实例”，防止误触。

4.2 模块二：月度课表首页（默认Tab - 日历视图）
展示：顶部月切换箭头，中间标准月日历格子。

格子标识：当天格子里用小圆点颜色表示该日课程整体财务状态（绿=全部已收，黄=存在未收，灰=无课）。

下方列表：点击某日期，下方动态展示该日所有课程卡片（按时间排序）。卡片信息包含：工作室名称、课程标签、开始-结束时间、红底白字显示“建议出门时间”。

卡片快捷操作：左滑卡片，出现“编辑”和“删除”按钮。

4.3 模块三：课程生成与复制（核心效率工具）
手动添加：点击日历页右下角“+”→ 弹窗选择“常规课”或“临时代课”。

选“常规课”时，要求选择“课程模板”（提前创建好）；

选“临时代课”时，要求填写工作室、时间、内容，不保存为模板。

月度复制（重点）：

在月切换处增加“复制上月排班”按钮。

执行逻辑：系统只遍历上月“课程模板”中类型=常规的条目，按新月份的具体日期对应星期，生成新的课程实例。

强制重置：新生成的实例，开课状态一律强制设为“待开”，收款状态一律强制设为“未收”。临时代课和上月停课记录绝不复制。

冲突提示：若新月份某天已存在课程实例，弹出列表让用户勾选“覆盖”或“跳过”。

4.4 模块四：一键批量标记（解决月结疲惫）
位置：日历页底部工具栏。

筛选前置：允许先筛选“全部工作室”或指定“某个工作室”。

三个独立大按钮（彼此无关联，按需点按）：

“标为正常开课”：将所有选中课程的开课状态设为“已开”（跳过“停课”状态）。
“标为停课”：将所有选中课程的开课状态设为“停课”。点击后强制弹窗，显示“是否自动按各工作室比例计算补偿？”（是→自动写入实际应收金额=单价×补偿比例，同时收款状态建议设为“已收”或弹窗询问）。
“标为课时费已发”：将所有选中课程的收款状态设为“已收”（即使开课状态是停课，也允许标记，符合你的业务）。
撤销：提供“撤销上一步批量操作”按钮，防止误标记。

4.5 模块五：对账中心（第二Tab）
视图：列表形式，按“工作室”分组，默认显示当前月份。

汇总行：每个工作室头部显示“总节数 | 应收合计 | 实收合计 | 未收差额”。

发薪日提醒：在该工作室头部右侧显示标签“预计X号发薪”，并用浅色进度条显示本月已过天数。

钻取详情：点击工作室名称，展开该工作室本月所有课程明细列表（显示日期、状态、金额）。

4.6 模块六：长期存档（零丢失）
自动存档：每月1号凌晨（或首次打开App时检测），自动将上个月全部数据（工作室+模板+实例）导出为JSON文件，存放于手机本地/Documents/Archives/目录。

恢复机制：设置页提供“从存档恢复”入口，选择JSON文件即可完全还原。

5. 非功能需求（独立开发者必须遵守）
项目	具体要求
性能	日历滑动、月份切换响应时间 < 300ms（SQLite索引建在课程实例.日期字段上）。
数据安全	所有数据仅存本地SQLite，不申请任何网络权限（iOS/Android安装包无需联网）。
备份	不依赖云，用户手动用手机自带的文件管理备份/Documents/目录即可。
UI风格	极简黑白灰底，仅用红/黄/绿三色作为状态强调色。无动画、无渐变、无花哨字体。
6. 用户核心操作路径（故事线）
月初第一天：点击“复制上月排班” → 确认覆盖 → 本月常规课全部生成。

日常上课：打开App看首页 → 看到“建议出门时间” → 出发。下课后点进该课卡片，将状态改为“已开”，顺便记一句音乐名。

临时代课：点击“+” → 选“临时代课” → 填完保存，仅当日显示。

遇到停课：点击该课卡片 → 点“设为停课” → 弹窗问是否算补偿 → 点“是”自动算钱。

月底对账：切到对账Tab → 扫一眼A工作室“应收是否等于实收” → 若没收到，记录后去催。

7. MVP开发优先级（P0必做 / P1可延后）
优先级	模块	原因
P0	工作室CRUD + 手动添加课程实例	没有它，连第一节课都记不了
P0	日历首页展示 + 点击查看当日详情	这是你每天打开App的核心动作
P0	三个一键批量标记	这是你月底解脱双手的命脉
P1	月度复制常规课	第二个月才能用，但逻辑必须提前设计好
P1	对账汇总列表（按工作室统计）	初期可用详情页手动加总，但报表最好第二周做出来
P2	JSON自动归档	等你用满一个月再写这个模块也不迟，先手动备份db文件应急
8. 边界与异常处理（防止App崩溃）
发薪日设置5/10号，显示时标注“（本月顺延）”。

停课补偿金额：保留2位小数（四舍五入），金额字段用REAL类型存储，不做高精度decimal以免复杂化。

删除工作室时：级联删除所有关联的模板和实例，操作前必须让用户勾选“我已知晓”复选框再点确定。
